Mission
Build an agent reputation system that tracks how agents interact with Sentinel over time. Returning agents with good history get faster, cheaper verification. Unknown agents get full-depth checks. This replaces the current model where every caller is treated identically regardless of history.

The system is aligned with ERC-8004 (Agent Trust Protocol), a proposed standard for on-chain agent identity and reputation. We don't implement ERC-8004 directly (the standard is still in draft), but we design our data model to be forward-compatible so we can adopt the standard when it finalizes.

Strategic Context: Anti-Overfit Guardrails
Reputation passes the net load test:
  - REDUCES: Trusted agents skip some checks (e.g., skip OFAC re-screening for a counterparty that was clean 2 hours ago)
  - REPLACES: Repeated verification depth with a reputation score
  - NO NEW SPOF: Reputation is stored in Redis + EAS (same infrastructure as Phase 0-1)

Architecture Overview

Three reputation tiers with different verification depths:

  UNKNOWN (default) — Full verification depth
    - All API calls executed
    - All checks at maximum depth
    - This is current behavior — no degradation

  RECOGNIZED — Moderate verification depth
    - Criteria: 5+ paid verifications, 0 flagged transactions in last 30 days
    - Benefits: Skip OFAC re-check if last check was <24h ago, reduced Etherscan depth (skip age re-computation if cached), preflight cache TTL extended to 10 min (from 5 min)
    - Net effect: ~30% fewer API calls per request

  TRUSTED — Minimum viable verification depth
    - Criteria: 20+ paid verifications, 0 flagged transactions in last 90 days, verified through ERC-8004 registry (when available)
    - Benefits: All RECOGNIZED benefits + skip contract metadata re-fetch if attestation <48h old, preflight cache TTL extended to 15 min, priority queue (future)
    - Net effect: ~50% fewer API calls per request

IMPORTANT: Reputation NEVER skips the final scoring step. It only reduces the NUMBER of upstream API calls by extending cache TTLs and reusing recent attestations. The scoring engine always runs with whatever data is available. A TRUSTED agent hitting a suspicious new contract still gets full scrutiny — reputation applies to the agent, not to the target.

Build 1: Agent Profile Storage (lib/reputation/store.js)

Agent profiles are stored in Redis with the following structure:

Key: sentinel:agent:{walletAddress}
Value (JSON):
  {
    wallet: "0x...",
    tier: "unknown",              // "unknown" | "recognized" | "trusted"
    total_verifications: 0,       // Lifetime count of paid verifications
    verifications_30d: 0,         // Rolling 30-day count
    flagged_transactions: 0,      // Count of verifications where agent's counterparty was UNSAFE
    last_verification: null,      // ISO timestamp
    last_ofac_check: null,        // ISO timestamp of most recent clean OFAC screen
    first_seen: "2026-04-02T...", // When this agent first used Sentinel
    tier_updated: null,           // When tier last changed
    metadata: {}                  // Reserved for ERC-8004 fields
  }

TTL: None (agent profiles persist indefinitely in Redis). If Redis data is lost, agents start over as UNKNOWN — this is acceptable.

Exports:
  - getAgentProfile(walletAddress) — Returns profile or creates default UNKNOWN profile
  - updateAgentProfile(walletAddress, verificationResult) — Increment counters, recalculate tier
  - getAgentTier(walletAddress) — Quick tier lookup (used on hot path)
  - recalculateTier(profile) — Pure function: given a profile, return the correct tier

recalculateTier logic:
  function recalculateTier(profile) {
    if (profile.total_verifications >= 20 &&
        profile.flagged_transactions === 0 &&
        profile.verifications_30d >= 5) {
      return "trusted";
    }
    if (profile.total_verifications >= 5 &&
        profile.flagged_transactions === 0) {
      return "recognized";
    }
    return "unknown";
  }

Note: The 90-day clean history requirement for TRUSTED is simplified to flagged_transactions === 0 for now. When we have enough data to track per-period flags, we can tighten this.

updateAgentProfile logic:
  - Increment total_verifications
  - Update last_verification timestamp
  - If the verification result contained UNSAFE verdict or OFAC flags → increment flagged_transactions
  - If the verification included a clean OFAC screen → update last_ofac_check
  - Recalculate tier
  - If tier changed, update tier_updated timestamp and log the change
  - Write back to Redis

Rolling 30-day count:
  - Simple approach: Store a Redis sorted set sentinel:agent:{wallet}:history with timestamps as scores. On each verification, ZADD the timestamp. On tier recalculation, ZRANGEBYSCORE to count entries in last 30 days. ZREMRANGEBYSCORE to prune entries older than 90 days.
  - This gives us accurate rolling windows without complex time-bucketing.

Build 2: Tier-Aware Cache TTLs (lib/reputation/tiers.js)

Export a function that returns cache TTLs based on agent tier:

  // NOTE: These base values MUST match the CACHE_TTL object in server.js (set up in Phase 0).
  // This function returns overrides — import the actual CACHE_TTL from server.js as the source of truth,
  // or read the values at runtime. Do NOT hardcode a second copy.
  function getTierCacheTTLs(tier, baseTTLs = CACHE_TTL) {
    const base = { ...baseTTLs };

    if (tier === "recognized") {
      return {
        ...base,
        preflight: 600,       // 10 min (extended)
        counterparty: 1800,   // 30 min (OFAC less frequent)
      };
    }

    if (tier === "trusted") {
      return {
        ...base,
        preflight: 900,       // 15 min (extended)
        counterparty: 3600,   // 1 hour
        contractMetadata: 172800, // 48h (extended)
      };
    }

    return base; // unknown
  }

Also export:
  function shouldSkipOfacRecheck(profile) {
    if (!profile.last_ofac_check) return false;
    const hoursSinceCheck = (Date.now() - new Date(profile.last_ofac_check).getTime()) / (1000 * 60 * 60);
    if (profile.tier === "recognized" && hoursSinceCheck < 24) return true;
    if (profile.tier === "trusted" && hoursSinceCheck < 48) return true;
    return false;
  }

Build 6: ERC-8004 Forward Compatibility

ERC-8004 (Agent Trust Protocol) proposes a standard for on-chain agent identity. Key concepts:
  - Agent Registry: A contract where agents register with their wallet + metadata
  - Trust Scores: On-chain reputation scores issued by trust providers (like Sentinel)
  - Delegation: Agents can delegate trust checks to services

We don't implement ERC-8004 now, but we prepare:

  1. In lib/reputation/store.js, the `metadata` field in the agent profile is reserved for ERC-8004 fields:
     metadata: {
       erc8004_registered: false,    // Will check on-chain registry when available
       erc8004_agent_id: null,       // Will be the on-chain agent ID
       erc8004_trust_score: null,    // Will be Sentinel's issued trust score
     }

  2. In lib/eas/config.js, add a comment block:
     // ERC-8004 COMPATIBILITY NOTES
     // When ERC-8004 finalizes, Sentinel will:
     // 1. Check the Agent Registry contract to verify agent identity
     // 2. Issue on-chain trust scores via the Trust Score interface
     // 3. Accept delegated trust checks from registered agents
     // 4. Map our tier system to ERC-8004 trust levels
     // Tracking: https://eips.ethereum.org/EIPS/eip-8004

  3. In the /agent/:walletAddress response, include erc8004_compatible: false in meta. This flag will flip to true when we integrate.

  - Do NOT implement ERC-8004 on-chain integration yet. Only prepare the data model.
  - Do NOT modify lib/scoring-engine/index.js.
